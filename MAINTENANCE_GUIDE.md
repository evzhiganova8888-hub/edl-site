# EDL · Maintenance Guide v1.0

**Версия:** 1.0 · 26 апреля 2026  
**Назначение:** как поддерживать сайт после релиза. Кто что может, что трогать нельзя, как делать типовые правки.

## 1. Кто что может менять

| Что | Кто | Как |
|---|---|---|
| Тексты на сайте (любые) | Любой в команде EDL | Прямая правка HTML через GitHub веб-интерфейс |
| Скриншоты системы | Любой в команде EDL | Замена PNG в корне репо |
| TG-витрина (3 поста) | Любой в команде EDL | Правка `index.html` блок 10 |
| Цены, сроки в продуктах | **Только Екатерина** + синхронизация в Источнике Правды | После обновления — правка в HTML |
| Логотип, цвета, типографика | **Никто без согласования.** Если есть веская причина — обсуждение с фаундером, обновление Brand Guidelines, потом код | См. `BRAND_GUIDELINES.md` |
| Структура страниц, добавление новых | По решению команды | Через PR с код-ревью |
| Apps Script, формы, аналитика | Технический администратор | Отдельный процесс |

## 2. Типовые правки — пошагово

### 2.1 Изменить текст на странице

1. Открыть нужный файл на GitHub (`index.html`, `sprint.html`, `diagnostic.html`, `404.html`)
2. Кнопка ✏️ карандаш справа сверху → редактировать
3. Найти нужный блок (они помечены HTML-комментариями: `<!-- ───── 01 · Hero ───── -->` и т.д.)
4. Изменить текст
5. Прокрутить вниз: `Commit changes`
6. Через 30–60 секунд изменения на сайте

### 2.2 Обновить скриншот системы в hero

1. Подготовить новый PNG (рекомендуемый размер: 600×750, обезличены клиенты)
2. На GitHub в корне репо: `Add file → Upload files`
3. Перетащить новый PNG, имя `screenshot-hero.png` (если такой файл уже есть — он перезапишется)
4. Commit

### 2.3 Обновить пост в TG-витрине

1. Открыть `index.html`, найти блок 10 (TG showcase)
2. Найти нужную карточку `<article class="tg-card">`
3. Заменить текст в `<div class="tg-card__body">`
4. Если хочется указать дату или короткую тему — заменить значение в `<span>` внутри `tg-card__meta`
5. Commit

### 2.4 Изменить цену продукта

⚠️ **Этот тип правки требует синхронизации с продажами.** Цена меняется не только на сайте, но и в скриптах продаж, прайс-листе, КП-шаблонах.

1. Сначала — обновить `EDL_Source_of_Truth_v2.md` и `EDL_Pricelist_May2026.md`
2. Потом — найти все места цены на сайте (поиск по репо: например, `180 000`)
3. Заменить во всех HTML-файлах
4. Прокатиться по странице — проверить, что нигде не осталось старого числа

### 2.5 Добавить FAQ-вопрос

1. Открыть `index.html`, найти блок 09 (FAQ)
2. Скопировать существующий блок `<div class="faq-item">...</div>` целиком
3. Вставить как новый
4. Заменить:
   - `data-faq-id="..."` на новый уникальный ID (например, `data-faq-id="new-question"`)
   - Текст в `<span>` внутри `<button class="faq-item__trigger">`
   - Текст в `<div class="faq-item__inner">`
5. Commit

## 3. Конвертация SVG → PNG (favicon, OG-image)

### 3.1 OG-image из `og-image.svg`

**Способ 1 — самый простой (через сервис).**

1. Открыть [cloudconvert.com/svg-to-png](https://cloudconvert.com/svg-to-png)
2. Загрузить `og-image.svg`
3. Кнопка ⚙️ справа от файла → Width: 1200, Height: 630, DPI: 144
4. Convert → Download
5. Переименовать в `og-image.png`
6. Загрузить в корень репо на GitHub

**Способ 2 — через Figma (для тех, кто им пользуется).**

1. Открыть Figma, новый файл
2. Перетащить `og-image.svg` на canvas
3. Выделить → Export → PNG 1x → 1200×630 → Download
4. Положить в репо

**Способ 3 — через Mac Preview.**

1. Открыть `og-image.svg` двойным кликом (откроется в Preview или Safari)
2. В Preview: Export → Format: PNG → размер 1200×630
3. Загрузить в репо

После конвертации **проверить превью** в [opengraph.xyz](https://www.opengraph.xyz) — ввести URL сайта, увидеть, как выглядит при шаринге.

### 3.2 Favicon-варианты из `favicon.svg`

Самый простой путь — все размеры за один заход:

1. [realfavicongenerator.net](https://realfavicongenerator.net)
2. Upload `favicon.svg`
3. Сервис генерирует полный набор: `favicon-16x16.png`, `favicon-32x32.png`, `apple-touch-icon.png`, Android-иконки, etc.
4. Скачать ZIP, распаковать
5. Положить в корень репо: `apple-touch-icon.png`, `favicon-32x32.png`, `favicon-16x16.png`
6. (Опционально) положить остальные файлы из ZIP-а — для полной совместимости со всеми устройствами

## 4. Что трогать нельзя без понимания последствий

### 4.1 Файл `styles.css` — раздел `:root` (CSS variables)

Изменение этих значений каскадирует **на весь сайт**. Не трогать без обсуждения с дизайнером.

```css
:root {
  --brand-navy: #1A1B2E;
  --bg-primary: #FAF9F6;
  --accent: #C8FF00;
  /* ... */
}
```

### 4.2 Файл `main.js` — переменная `METRICA_ID`

Если её случайно стереть или испортить — аналитика перестанет работать. Менять только при перевыпуске счётчика Метрики.

### 4.3 Файл `CNAME`

Содержит одну строку: `elephantdreams.ru`. Менять только если меняется домен. Убрать → сайт перестанет открываться по `elephantdreams.ru`.

### 4.4 SVG-логотипы (`logo.svg`, `favicon.svg`, `logo-wordmark.svg`)

Менять только по решению фаундера. Если меняется — обновить:
- Файл-источник
- Все упоминания в HTML (header, favicon links)
- `BRAND_GUIDELINES.md`

### 4.5 Структура HTML на главной

Блоки на главной странице — это срежиссированный путь пользователя (Hero → Proof → Authority → Architecture → Funnel → ICP → Founder → No → FAQ → TG → Final CTA). Менять порядок — это менять режиссуру. Делать только после обсуждения.

## 5. Версионирование

При значительной правке (новая страница, существенное изменение копи, замена скриншотов) — обновлять версию в `README.md` и в `BRAND_GUIDELINES.md` (если применимо).

| Тип правки | Бамп версии |
|---|---|
| Опечатка, мелкая правка текста | Не бампим |
| Переписан целый блок | Patch (1.0 → 1.0.1) |
| Добавлена страница, новый продукт | Minor (1.0 → 1.1) |
| Ребрендинг, смена позиционирования | Major (1.0 → 2.0) |

## 6. Деплой и rollback

### 6.1 Каждый коммит = деплой

GitHub Pages автоматически перерезирует сайт при каждом push в `main`. Нет ручного «деплоить»-шага. Через 30–60 секунд после коммита изменения уже на сайте.

### 6.2 Откатиться к прошлой версии

Если что-то пошло не так:

1. На GitHub: вкладка `Commits` (на главной репо)
2. Найти прошлый рабочий коммит, кликнуть
3. На странице коммита: `Browse files` (вверху справа)
4. Найти проблемный файл, скопировать его содержимое
5. Вернуться в текущую версию, заменить файл, commit

Альтернативно — через `git revert HEAD` если работаете локально через GitHub Desktop.

### 6.3 Ничего не открывается после правки

Самые частые причины (по убыванию):
1. Сломанный синтаксис HTML (незакрытый тег) — проверить через [validator.w3.org](https://validator.w3.org)
2. Ошибка в JS — открыть Chrome DevTools → Console, посмотреть красные строки
3. Кэш браузера — Cmd+Shift+R (force-refresh)
4. GitHub Pages ещё не обновился — подождать 1–2 минуты
